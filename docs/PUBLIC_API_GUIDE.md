# NocoDB Public APIs, SDK, and Component Guide

This guide consolidates the public interfaces shipped in this repository and points you to the autogenerated references that live alongside the code.

- **REST API endpoints** → exhaustive, auto-generated reference in `docs/REST_API_REFERENCE.md`.
- **TypeScript SDK** → everything exported from `packages/nocodb-sdk`.
- **CLI tooling** → `packages/nc-cli`, distributed as the `xc` binary.
- **Embeddable GUI assets** → `packages/nc-lib-gui`.

## REST API Overview

- OpenAPI schema source: `packages/nocodb/src/schema/swagger-v2.json`.
- Generated documentation: `docs/REST_API_REFERENCE.md` (run `node scripts/generate-rest-docs.mjs > docs/REST_API_REFERENCE.md` to refresh from schema changes).
- **Auth flow**: call `POST /api/v1/auth/user/signin`, capture the `token` field, then pass it as the `xc-auth` header (`xc-auth: <token>`). API tokens can be created via `POST /api/v2/meta/bases/{baseId}/api-tokens`.
- **Base + table management**: `/api/v2/meta/bases/{baseId}/tables` and `/api/v2/meta/bases/{baseId}/{sourceId}/tables` provide CRUD for tables, columns, and views.
- **Data APIs**: `/api/v2/tables/{tableId}/records` (rows), `/api/v2/tables/{tableId}/links/{columnId}` (link records), `/api/v2/public/...` (shared/public views).

### Example: end-to-end cURL session

```bash
BASE="p_124hhlkbeasewh"
TABLE="tbl_mk3yy0i"* # replace with your table ID
HOST="https://your-nocodb.example.com"

# Sign in and keep the xc-auth token
TOKEN=$(curl -s -X POST "$HOST/api/v1/auth/user/signin" \
  -H "Content-Type: application/json" \
  -d '{"email":"user@example.com","password":"Password"}' | jq -r .token)

# Create a column
curl -X POST "$HOST/api/v2/meta/tables/$TABLE/columns" \
  -H "xc-auth: $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"title":"Status","uidt":"SingleSelect","colOptions":{"options":["Todo","Doing","Done"]}}'

# Insert a row
curl -X POST "$HOST/api/v2/tables/$TABLE/records" \
  -H "xc-auth: $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"Status":"Todo","Title":"Draft API doc"}'

# Query the rows with filters and sorts
curl "$HOST/api/v2/tables/$TABLE/records?sort=Status&where=%7B%22Status%22%3A%22Todo%22%7D" \
  -H "xc-auth: $TOKEN"
```

Refer to `docs/REST_API_REFERENCE.md` for every available route, parameters, and response payload.

### UI schema & component helpers

- `UITypes` and helpers (`isNumericCol`, `isVirtualCol`, `numericUITypes`, etc.) describe every supported field type in Grid/Form/Kanban views. They are the backbone for client-side rendering decisions and validation.
- `XcUIBuilder` exports the `XcForm`, `XcItem`, `XcType`, and `XcActionType` primitives for describing settings forms or plugin installation UIs.
- `columnRules/*` declares rule sets for special field types (e.g. QR/barcode columns).
- `sqlUi/*` exports per-database SQL helpers (UI metadata for Databricks, MSSQL, MySQL, Postgres, Snowflake, Oracle, SQLite).

### Data + formula helpers

- `helperFunctions.ts` – utilities to work with metadata (`filterOutSystemColumns`, `isSelfReferencingTableColumn`, `extractRolesObj`, role serialization, derived rollup options, attachment filename deduping).
- `aggregationHelper.ts` – enumerations of supported aggregations and `getAvailableAggregations(type, parsedFormula)` to drive summary rows.
- `formulaHelpers.ts` – jsep-powered parsing, validation, and metadata for every built-in formula (syntax, docs URLs, custom validations). Use `substituteColumnAliasWithIdInFormula` to canonicalize formulas and `formulas[FN_NAME]` to render contextual help.
- `form/index.ts` – base interfaces for rendering shareable forms (field layout, action definitions).

### SDK usage patterns

**Creating a base, syncing schema, and inserting rows**

```ts
import { Api } from 'nocodb-sdk';

const client = new Api({ baseURL: NOCODB_URL, securityWorker: () => ({ headers: { 'xc-auth': token } }) });

const base = await client.base.create({ title: 'Product Ops' });
const source = await client.source.create(base.id, { type: 'pg', config: {...} });
await client.source.sync(base.id, source.id);
const table = await client.dbTable.create(base.id, source.id, {
  title: 'Tasks',
  columns: [{ title: 'Title', uidt: 'SingleLineText' }],
});
await client.dbDataTableRow.create(table.id, { Title: 'Draft API doc' });
```

**Building dynamic filters**

```ts
import { filterOutSystemColumns, getAvailableAggregations } from 'nocodb-sdk';

const columns = await client.dbTableColumn.list(table.id);
const visibleCols = filterOutSystemColumns(columns.list);
const aggregations = getAvailableAggregations(visibleCols[0].uidt);
```

## CLI (`packages/nc-cli`)

The CLI (binary: `xc`) scaffolds NocoDB projects and can provision Docker or Node deployments.

Key entrypoint: `packages/nc-cli/src/index.ts` wires `yargs` parsing, update notifications, and delegates to `CliMgr.process`.

Supported public commands (`CliMgr.process`):

- `xc man [command]` – show contextual help (`Util.showHelp`).
- `xc new <projectName>` – interactive wizard implemented in `NewMgr.getNewProjectInput`:
  - Prompts for metadata DB type/credentials.
  - For Node projects, downloads `nocodb-seed` starter, writes `.env`, installs deps, and launches `npm run start`.
  - For Docker, prints or executes the `docker run nocodb/nocodb:latest` command with the constructed `NC_DB` connection string.

Example workflow:

```bash
xc new my-nocodb
# choose Postgres → provide credentials
# pick "As Docker" to have the CLI print the docker run command with NC_DB pre-filled
```

Internals worth noting:

- `NewMgr` (see `packages/nc-cli/src/lib/mgr/NewMgr.ts`) handles prompts, connection validation (`tcp-port-used`), and environment file generation.
- `Client` sends scaffold requests back to the hosted service when code generation is involved.
- `util/Util.ts` exposes wrappers for shell execution, help text, and string helpers consumed throughout the CLI.

## Embeddable GUI middleware (`packages/nc-lib-gui`)

`packages/nc-lib-gui/lib/XcLibGui.js` exports a minimal Express middleware that serves the compiled frontend assets (`nc-gui` build output copied via `npm run build:copy`).

```1:11:packages/nc-lib-gui/lib/XcLibGui.js
const express = require('express');
const path = require('path');

class XcLibGui {
  static expressMiddleware(dashboardPath) {
    const router = express.Router();
    router.use(dashboardPath, express.static(path.join(__dirname, 'dist')));
    return router;
  }
}
```

Usage example inside a custom backend:

```ts
const express = require('express');
const XcLibGui = require('nc-lib-gui/lib/XcLibGui');

const app = express();
app.use(XcLibGui.expressMiddleware('/dashboard'));

app.listen(8080, () => console.log('Dashboard available at http://localhost:8080/dashboard'));
```

## Regenerating and extending the docs

1. Update the OpenAPI schema (`packages/nocodb/src/services/api-docs/swagger/**/*.json`), then run `node scripts/generate-rest-docs.mjs > docs/REST_API_REFERENCE.md`.
2. Extend this overview (`docs/PUBLIC_API_GUIDE.md`) whenever new SDK modules, CLI commands, or embeddable components become public.

These two files give contributors and integrators a high-level guide plus the exhaustive REST endpoint matrix sourced from the codebase itself. Use them as the single source of truth for public surface area discussions.
